## 下流工程と代表成果物

1. 詳細設計: 要件を実装可能な仕様へ落とし込み、詳細設計書を作る。
2. 構築: プログラムとデータベースを実装する。
3. テスト: 単体、結合、総合の順に確認し、テスト報告を残す。
4. ユーザー教育: 操作・運用に必要な教育を行い、教育報告を残す。
5. 導入・検収: システムを運用側へ引き渡し、完了報告と検収を行う。

## 問題から導く実務ルール

| 領域 | 典型問題 | 実務ルール |
|---|---|---|
| 人的資源 | 人の入れ替わりで仕様理解が遅れる | メンバー変更、学習、説明、レビューによる一時的な生産性低下を予定へ反映する |
| 時間 | 仕様変更が積み重なり納期へ影響する | 変更の納期・品質・コスト影響を示し、承認後に受け入れる |
| 品質 | 本番データ量でDB性能が出ない | 本番に近いデータ量・構成で性能を計測する |
| 品質 | 遠隔地からの応答が悪い | 回線速度、経路、遅延を想定し、配備場所と構成を検討する |
| 労働 | 残業と休日出勤が常態化する | 集中力低下と欠陥増加を管理リスクとして扱い、範囲・計画を是正する |
| 技術 | 新技術の未知の不具合で開発が止まる | 実績ある技術を優先するか、小さな事前検証後に採用する |

## 小規模・AI開発への読み替え

- 詳細設計書は、変更対象・入出力・例外・受入条件を記した短い実装計画で代替できる。
- テスト報告書は、実行コマンド、結果、未確認事項を残した作業報告で代替できる。
- 教育報告書は、利用者が少ない場合、確認済みの操作手順と運用手順で代替できる。
- ステアリング・コミッティは、個人開発では「変更を今入れるか後に分けるか」の明示的な意思決定へ読み替える。
- AIエージェントの追加も、文脈共有、出力レビュー、統合作業が必要な人的資源の追加として扱う。
