# 下流工程プロジェクトガード

大学講義の下流工程管理を、小規模アプリ開発で実行できる品質ゲートへ変換する。書類の量ではなく、成果物と検証証拠で判定する。

## 原則

- 現在の工程、変更範囲、利用者影響を先に特定する。
- 事実を `確認済み`、`未確認`、`対象外` に分ける。証拠のない成功判定をしない。
- 小規模・個人開発では最小限の成果物に縮約する。低リスク変更へ一律に重い工程を課さない。
- 納期・品質・範囲の衝突を隠さない。仕様を増やす場合は、納期延長・品質リスク受容・別リリース化のいずれかを明示する。
- 残業を恒常的な遅延対策にしない。作業削減、順序変更、自動化、期限または範囲の再交渉を優先する。
- 講義由来の判断原則が必要な場合は [references/lecture-principles.md](references/lecture-principles.md) を読む。

## 1. リスク強度を決める

変更を次のいずれかに分類する。

- `軽微`: 文言、見た目、局所的で可逆な修正。関連テストと目視確認を行う。
- `標準`: 機能追加、API・DB・状態管理の変更。単体・結合・主要ユーザーフローを確認する。
- `高リスク`: 認証、決済、個人情報、権限、データ移行、外部公開、不可逆操作、基盤変更。ロールバック、セキュリティ、バックアップ、本番相当テストまで要求する。

判断できない場合は一段高く扱い、理由を書く。

## 2. 工程と成果物を対応させる

| 工程 | 最小成果物 | 通過条件 |
|---|---|---|
| 詳細設計 | 変更対象、入出力、例外、受入条件 | 実装とテストが同じ条件を参照できる |
| 構築 | コード、設定、DB変更 | 静的検査と対象テストが成功する |
| 単体テスト | 関数・コンポーネント単位の検証 | 正常系、境界値、主要異常系を確認する |
| 結合テスト | API、DB、外部サービス間の検証 | 契約、状態遷移、失敗時の処理を確認する |
| 総合テスト | 利用者の主要フローの検証 | 本番に近い構成・データ量・通信条件で成立する |
| 導入・検収 | リリース記録、操作説明、運用手順 | 受入条件、監視、復旧・ロールバックが確認できる |

工程名にこだわらず、変更リスクに必要な成果物だけを作る。

## 3. 5つの監査を行う

### 変更管理

1. 当初範囲との差分を列挙する。
2. コード、DB、API、テスト、納期、運用への影響を見積もる。
3. 今回入れる、次回へ分ける、却下する、のいずれかを決める。
4. 顧客案件では、追加作業を着手前に選択肢と影響付きで承認へ回す。

### 人的資源

- 新しい担当者やAIエージェントを追加しても、直後に速度が上がると仮定しない。
- 仕様理解、説明、レビュー、統合のコストを計画へ入れる。
- 担当、レビュー責任、引き継ぎ情報を明示する。
- 長時間労働による一時的な帳尻合わせを進捗として評価しない。

### 技術選定

- 安定した既存技術で要件を満たせるかを先に確認する。
- 新技術を選ぶ場合は、採用理由、未知のリスク、撤退条件を記録する。
- 小さなスパイクでビルド、主要API、配備、性能、デバッグ可能性を検証してから中核へ採用する。
- 「新しい」「速い」という宣伝だけを採用根拠にしない。

### 品質・性能

- テスト結果をコマンド、ログ、スクリーンショット、計測値などの証拠に結びつける。
- DB性能は本番相当の件数、インデックス、クエリで測る。
- 地理的に離れた利用者がいる場合は、帯域・遅延・タイムアウト・再試行を検証する。
- 外部サービス障害、空データ、重複送信、権限不足などの失敗経路を確認する。

### 導入・運用

- 操作手順は「画面の使い方」、運用手順は「業務でいつ誰が何をするか」と分ける。
- リリース前にバックアップ、移行、監視、問い合わせ先、ロールバックを確認する。
- 検収条件を実装後に作らない。着手前の受入条件と照合する。

## 4. 作業中の適用タイミング

- 実装前: リスク強度、受入条件、必要テストを決める。
- 実装後: 差分に応じた単体・結合テストを実行する。
- 大きな節目: 主要ユーザーフローと性能・運用リスクを確認する。
- リリース前: 未確認事項とロールバック可否を示し、Go / Conditional Go / No-Go を判定する。
- 障害後: 原因だけでなく、どの工程のゲートで検出すべきだったかを記録する。

## 5. 出力する

原則として次の形式で簡潔に返す。

```md
## 判定
- リスク強度:
- 現在工程:
- リリース判定: Go / Conditional Go / No-Go / 未判定

## 確認証拠
- 確認済み:
- 未確認:
- 対象外:

## 主要リスク
| 優先度 | リスク | 影響 | 対策 | 完了条件 |

## 次の品質ゲート
- 実行項目:
- 必要な証拠:
- 通過条件:
```

実装依頼では報告だけで止めず、権限内で修正と検証まで行う。監査・診断依頼では、ユーザーが変更も求めていない限りコードを変更しない。
